# Learnings

- `SettingsSwitcherPage` now uses a single interlocked busy gate plus a shared `SetSwitcherBusy` helper so install/remove/verify work cannot be re-entered and controls are always restored in `finally`.
- Windows package install/remove failures can be captured from WinRT `DeploymentResult` with `ActivityId`, `ErrorText`, and `ExtendedErrorCode`, then wrapped in a launcher-facing exception for structured logging.
- A scriptable switcher QA hook can live in `App.xaml.cs` behind a dedicated CLI flag while reusing extracted page logic; `launcher.log` plus process exit code is a workable verification surface even for the WinExe launcher.
- The toast lifecycle is more reliable when `MessageBox` owns its own close task and animation completion, while `MainWindow` serializes creation so rapid reuse leaves only the latest transient notification active.
- A bounded `--verify-notifications` startup path can exercise rapid replacement plus manual and timed dismissals without manual WPF clicking by hosting the same notification flow inside an off-screen window.
- The notification verifier must be scheduled back onto the dispatcher as an `async void` startup callback; synchronously blocking the WPF startup thread or trying to null `StartupUri` at runtime prevents the verification path from completing cleanly.
- Launcher settings are more reliable when `src` pages persist immediately through a shared `Settings.TrySave(...)` path instead of waiting for window shutdown, because tray-minimize and crash paths otherwise leave the UI ahead of disk state.
- `Settings.Current` now recovers unreadable JSON by moving the bad file aside, logging the recovery, and sanitizing missing enum/string/dictionary values before restart-time consumers read them.
- A bounded `--verify-settings-persistence` startup path can prove round-trip persistence plus corrupt-file recovery by relaunching the WinExe in child verification stages and asserting the same startup snapshot logic used by `MainWindow`.
- Version feed rows need two-stage version handling: normalize remote versions to the launcher's three-part display/install format, but keep the optional fourth segment for deterministic duplicate preference and latest ordering when feeds emit multiple revisions of the same normalized build.
- Version catalog resilience is more reliable when each remote source is loaded independently, malformed rows are skipped with logging, and normalized ordering/latest selection are computed from the surviving per-platform entries instead of assuming every feed parses cleanly.
- Download integrity is safer when `HttpService.DownloadAsync(...)` streams into a random `.partial` file and only replaces the destination after the full payload and optional `Content-Length` check succeed, so failed downloads do not truncate the last good DLL or updater payload.
- Proxy/DoH coexistence is more truthful when `HttpServiceHandler` skips the launcher-side DoH rewrite while proxy mode is enabled and falls back to the original hostname if the Cloudflare lookup fails, avoiding false download failures caused by the resolver path itself.
- Self-update recovery is easier to understand when the updater helper writes a one-shot status marker and the relaunched launcher consumes it on startup to surface either a success toast or a locked-file retry message instead of silently restarting into an ambiguous state.
- Package replacement is safer when the switcher classifies each target as fresh install, reinstall, upgrade, downgrade, or platform switch and then uses one explicit sequence: backup existing data if Minecraft is installed, uninstall the current package, install the target package, and verify the resulting registered version/platform before claiming success.
- Active-launcher regression closure found two integrated UI gaps: the main window still allowed close/launch during version switch or backup restore work, and the version label still fell back to a fake `0.0.0 (UWP)` state when no package was registered.
