# Decisions

- Deployment failure classification stays launcher-side in `SettingsSwitcherPage` while low-level WinRT deployment details are captured in `lib` via `DeploymentOperationException`; this keeps user messages bounded without losing structured diagnostics.
- Backup restore guidance is only appended when the old package was removed before the switch failed, matching the backup-based recovery policy for failed reinstalls.
- Switcher failure mapping and busy-state gating were extracted into shared internal logic so both the WPF page and a new `--verify-switcher` startup path exercise the same behavior instead of duplicating test-only code.
- Transient launcher notifications now follow a latest-wins policy in `MainWindow`, dismissing any in-flight toast before showing the next one so rapid reuse cannot leave stacked stale visuals behind.
- `MessageBox` close completion is now event-driven instead of delay-driven so timer-based and manual dismissal share the same removal contract.
- App startup now creates `MainWindow` manually instead of relying on `StartupUri`, which lets `--verify-notifications` bypass normal window startup and shut down with a deterministic exit code after its dispatcher-run verification callback finishes.
- Settings persistence now treats malformed JSON as recoverable launcher state: the bad file is renamed to a timestamped `.corrupt` copy, defaults are restored in memory, and the recovery is logged instead of being silently swallowed.
- Restart-time settings verification was kept launcher-side by extracting a small `MainWindow.ReadStartupSettings(...)` snapshot and driving it from `--verify-settings-persistence`, avoiding test-only UI clicks while still exercising the same startup readers used in production.
- Switcher latest-version messaging now trusts `VersionCatalog.LatestUwpVersion` and `VersionCatalog.LatestGdkVersion` instead of recomputing from UI rows so degraded feeds and duplicate normalization cannot create contradictory latest claims in the page.
- Version feed hardening stays centered in `lib` by normalizing and validating feed rows before they enter `VersionCatalog`, while `src` only consumes the catalog's explicit platform/latest metadata and a dedicated `--verify-version-catalog` startup hook for scriptable degraded-source checks.
- Network/download hardening stays in `lib` by making `HttpService.DownloadAsync(...)` atomic and making `HttpServiceHandler` degrade to the original request when DoH resolution is unavailable, while `src/MainWindow.xaml.cs` only maps the resulting exceptions into truthful launcher status text and dialogs.
- Self-update lock handling stays in the existing batch-based updater architecture, but the helper now retries for a bounded window, records a startup status file, and relaunches the launcher with that status so `src` can report whether the replacement actually happened.
- Package replacement semantics are now centralized in `SettingsSwitcherLogic.SwitcherPlan`, so both UWP and GDK replacements follow the same launcher-side policy and failure mapping can distinguish a newer package still installed before removal from a lingering conflict reported after removal.
- Active launcher busy-state closure stays in `src/MainWindow.xaml.cs`: switch and backup-restore flows borrow a shared launcher busy scope so the main launch button and window close path stay blocked until the integrated operation finishes, while the version label now reads from `Minecraft.GetInstallState()` to report `Not installed` instead of a fake platform/version fallback.
